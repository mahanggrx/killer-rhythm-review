# 项目目标

这是一个用于游戏系统策划作品集的“杀手节奏复盘反馈”网页原型。

用户选择、导入一份人工制作的模拟对局事件日志，或按核心指标生成一份匿名合成日志后，系统验证并规范化日志，计算接敌、追击、发电机控制和永久减员结果指标，再通过确定性规则输出一个主要节奏断点和一条练习建议。若证据不足或没有明显断点，系统必须允许输出“未发现明确主要断点”，不得强行归因。

本项目是赛后学习与失败归因工具，不是玩家评分系统、真实战绩分析器、游戏平衡结论或局内辅助工具。

## 权威资料与规则基线

- 产品目标、用户问题和反馈原则以 `《黎明杀机》系统拆解与优化分析报告_最终版.docx` 为依据。
- 游戏机制、事件边界和状态转换以 `research/mechanics_research_2026-07-14.md` 为依据。
- 两份资料冲突时：游戏机制事实采用机制研究文件，产品意图采用分析报告。
- 第一版规则集固定为 `base_only_1v4_10.0.2`。
- 报告中的 75 秒、10% 等数值只是原型假设，不得描述为官方标准、真实平衡结论或经过数据验证的阈值。
- 不得静默支持规则集以外的技能、附加品、特殊杀手力量、替代挂钩机制、活动模式或 2v8。日志包含未支持机制时，必须警告或拒绝分析。

# 技术约束

- 使用 Vite、React、TypeScript。
- 使用 Vitest 编写测试；需要组件测试时可使用 React Testing Library 和 jsdom。
- 使用普通 CSS，不使用 Tailwind 或大型 UI 组件库。
- 优先使用 React 内置状态，不引入 Redux 等额外状态管理框架。
- 不使用后端、数据库、登录系统、云服务或远程持久化。
- 不连接、读取、抓取或修改真实游戏客户端、进程、内存、网络包、录像或玩家账号数据。
- 不使用机器学习。
- 不调用官方或第三方玩家数据接口。
- 中文界面；TypeScript 标识符和文件名使用清晰的英文命名。
- 项目必须可以完全静态构建并从 `dist/` 部署。
- 保持依赖克制；新增运行时依赖前，先确认原生 React/TypeScript/CSS 是否已经足够。

# 范围控制

- 第一版只分析接敌、追击、发电机控制和永久减员结果四个维度。
- 不实现完整对局回放、实时战况、地图坐标、板窗资源模拟、全部杀手/技能/附加品数据库、MMR、胜率或构筑推荐。
- 建议以玩家下一局可练习的行为为中心，不推荐具体技能、附加品、杀手或唯一正确打法。
- 不使用《黎明杀机》的受版权保护图片、图标、音频或界面素材；允许制作原创的暗色数据面板视觉。
- 所有示例日志必须是匿名、人工构造的合成数据。
- 内置生成器只用于构造规则测试日志，不得描述为真实对局数据还原、客户端日志导出或对玩家行为的完整模拟。

# 数据与机制建模约束

## 通用事件

- 每个事件必须具有 `eventId`、`timestampMs`、`eventOrder` 和 `type`。
- `timestampMs` 表示时间，`eventOrder` 决定相同时间戳下的严格先后顺序。
- 事件不得依赖数组当前顺序作为唯一时序依据；规范化后必须稳定排序。
- 对局结束时，未闭合的追击和回退区间必须以策略生成事件关闭，并标记 `censored: true`。

## 接敌与追击

- 第一版不使用人工 `target_acquired`；基础 schema 出现该事件时必须作为未知事件拒绝，不能把气场、尖叫、直视、杀手本能、足迹、血迹或声音自动等同于正式追击。
- 接敌指标只使用游戏正式 `chase_start` 作为有效接敌代理，并明确它不等于玩家主观发现目标的准确时刻。
- 正式追击必须以独立 `chaseId` 建模。
- `chase_start.source` 在基础规则集中只能是 `game_state`。
- 允许多个追击区间短暂并发，不得使用一个全局 `currentTarget` 覆盖全部追击。
- 主动放弃、目标丢失和转火属于分析口径；无法确认时使用 `unknown`，不得由系统猜测玩家意图。
- 若倒地时缺少正式 `chase_end`，允许按已声明的原型策略补充结束事件，但必须标记 `policyGenerated: true`。

## 发电机

- 发电机不能建模为单一互斥状态。
- 必须区分活动状态、受损标志、封锁标志、完成终态，以及回退的活跃、暂停、恢复和停止。
- 即时进度损失、持续回退和封锁必须分别记录。
- 同一次杀手行为产生的多个控机效果共享 `interferenceId`；统计干扰次数时按因果触发去重。
- 逃生者停止修理本身不算杀手有效控机。
- 只有活跃回退区间计入回退时长；因修理或封锁暂停的时间不得计入。
- 发电机完成后为终态，后续普通修理、破坏、回退或封锁生效事件属于语义错误。

## 逃生者、挂钩与结果

- 逃生者至少拆分为 `healthState`、`custodyState`、`controller`、`hookStage` 和 `outcome`，不得用单一枚举混合所有状态。
- `controller_changed` 表示玩家掉线后由 BOT 接管；它不等于角色离场或杀手永久减员。
- 自然从挂钩第一阶段进入第二阶段不是新的有效挂钩事件。
- 第一版不依据挂钩集中度、再次上钩转化或普通挂钩链形成时间判断玩家打法优劣。
- 献祭、处决和流血死亡属于永久减员；逃脱和 BOT 接管不算杀手减员。
- 第一版不模拟挂钩阶段转移；日志出现此类机制时应标记未支持，而不是伪装成普通挂钩流程。

# 架构约束

- 日志校验、规范化、指标计算、规则评估、主断点选择和时间线构建必须位于 React 之外。
- 指标计算必须是无副作用的纯函数；不得读取浏览器状态、组件状态或全局可变对象。
- 规则判断必须与 React UI 分离。
- React 组件不得直接计算核心指标、解释事件链或复制规则条件。
- UI 只消费稳定的分析结果模型，例如 `AnalysisResult`。
- 所有默认阈值集中存储在独立配置模块，不得散落在组件、样式或建议文案中。
- 阈值配置、指标定义、规则定义和建议文案应保持职责分离。
- 日志或阈值变化后，通过统一分析入口重新计算派生结果；不要把可推导数据重复保存为 React 状态。
- 相同日志、规则集和阈值必须始终产生相同结果。
- 不得使用随机数、当前时间或对象遍历偶然顺序影响主断点选择。
- 合成日志生成必须位于 React 之外并保持确定性；生成结果必须重新经过现有日志校验和指标回算，回算值与输入不一致时不得交给分析界面。

# 校验与容错约束

- 输入必须先经过运行时校验，不能依赖 TypeScript 类型代替 JSON 校验。
- 校验分为两层：
  - 结构错误：字段缺失、类型错误、ID 引用无效、数值越界等，阻止分析。
  - 语义警告或错误：事件乱序、区间未闭合、状态转换异常、样本不足、未支持机制等，按明确策略规范化、警告或拒绝分析。
- 缺失、重复、乱序、同时间戳和未完成事件不得导致页面崩溃、无限循环、`NaN` 或 `Infinity`。
- 无法计算的指标使用明确的不可用状态和原因，不得伪装为 0。
- 所有平均值必须处理零分母和删失样本。
- 任何自动修复或策略生成事件都必须可以追溯，不得静默改变原始含义。

# 指标与规则约束

- 平均追逐空窗统一纳入 `trial_start` 到首次 `chase_start`，以及全部并发追逐结束后到下一次 `chase_start` 的每个完整空窗；删失结束和对局结束后的尾段不形成样本。
- 首追时长固定定义为首个 `chase_start` 到同一 `chaseId` 的非删失 `chase_end`；不得以倒地、抱起或挂钩代替追逐结束。
- 首次击倒可以作为辅助追击指标，但包含搬运或挂钩的时间不得描述为首追时长。
- 高进度发电机干扰以干扰前进度为判断依据；高进度 episode 必须能够处理多次跨越阈值。
- 回退时长只累计活跃区间；覆盖率分母必须在指标定义中明确，不得使用含糊的“总时间”。
- 首次永久减员结果以首次献祭、处决或流血死亡发生时尚未完成的基础修理目标数表示；基础 1v4 规则需要完成 5 台发电机。
- 同时间戳下必须使用 `eventOrder` 判断发电机完成与首次减员的严格先后；没有永久减员时不得伪造数值。
- 第一版候选断点固定为：首次追逐持续较长、首次永久减员形成较晚、有效接敌空窗较长。高进度控机指标保留为解释数据，不单独形成主要候选。
- 规则引擎还必须支持 `no_clear_breakpoint`，证据不足时不得强行选择问题。
- 多条规则同时触发时，按证据充分性、相对阈值偏离、规则置信度、规则优先级和稳定 ID 顺序确定唯一结果。
- 内部可以使用 `severity` 比较候选，但不得将其作为玩家分数展示。
- 每个候选和最终断点必须保留触发规则、关联指标及 `evidenceEventIds`，保证页面能够解释“为什么得出该结论”。

# 产品与界面约束

- 默认只展示一个主要节奏断点，或“未发现明确主要断点”。
- 默认最多展示三个与该断点直接相关的关键指标和一条建议。
- 反馈使用中性、可行动的“建议”，不使用分数、评级、批评、羞辱或绝对归因文案。
- 使用“可能”“本局数据显示”等限定语，避免声称系统知道玩家主观意图。
- 详细数据默认折叠。
- 所有默认阈值标注为“原型待验证数值”。
- 必须能够从结果跳转或定位到支撑提示的时间线证据。
- 校验错误、数据警告、样本不足、删失区间和未支持机制必须清楚呈现。
- 阈值修改后结果实时重新计算，并支持恢复默认值；无效阈值不得进入分析引擎。
- 优先保证信息层级、键盘可用性、中文可读性和窄屏基本可用，再增加视觉装饰。

# 示例数据与测试约束

- 至少规划以下合成日志：均衡无断点、接敌空窗过长、首次追逐过长、首次永久减员较晚、多断点竞争、BOT 接管、流血减员、删失区间、非法日志。
- 每份有效样例必须有人工计算的预期指标和预期主断点，作为测试依据。
- 关键领域模块必须有单元测试：
  - 运行时校验与语义校验；
  - 稳定排序和区间规范化；
  - 四类指标计算；
  - 多规则竞争和确定性排序；
  - 合成日志输入边界、生成确定性、运行时校验和指标回算一致性；
  - `no_clear_breakpoint`；
  - 阈值边界、零分母、BOT、同时间戳、并发追击和删失区间。
- 至少有一条组件或集成测试覆盖“选择/上传日志 -> 校验 -> 分析 -> 修改阈值 -> 结果更新”的主流程。
- 修复缺陷时必须先补充或同步增加能够复现问题的测试。

# 工作方式与文件纪律

- 未经用户明确要求，不修改原始分析报告和机制研究文件。
- 不得静默改变已经确认的事件口径、指标公式或规则优先级；如需改变，必须同步更新说明、样例和测试。
- 先完成最小可验证纵切，再扩展界面或增加指标，避免一次性搭建与目标无关的框架。
- 保留用户已有修改，不覆盖无关文件，不执行破坏性 Git 或文件操作。
- 规划任务只输出方案；用户要求开始实现后才创建项目、安装依赖或写代码。

# 质量门与完成标准

项目初始化后，`package.json` 应提供稳定的非交互命令：

- `npm run test`：执行一次完整 Vitest 测试，不进入 watch 模式。
- `npm run typecheck`：执行 TypeScript 类型检查。
- `npm run build`：执行生产构建。

每个有意义的实现批次完成后必须：

1. 运行相关测试；
2. 在交付前运行完整 `npm run test`；
3. 运行 `npm run typecheck`；
4. 运行 `npm run build`；
5. 总结新增或修改的文件；
6. 报告仍然存在的问题、假设、警告和未覆盖范围。

如果项目尚未初始化、命令尚不存在或环境阻止运行，不得为了满足形式伪造通过结果。应明确说明未运行的检查、原因和下一步。在只有文档或规则文件变更且尚无前端项目时，不需要擅自初始化项目来执行测试与构建。
