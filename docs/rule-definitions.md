# 节奏断点规则口径（第一版）

适用规则集：`base_only_1v4_10.0.2`。

规则引擎接收已经计算完成的 `MatchMetrics`，输出唯一 `primaryFeedback` 和按确定性顺序排列的 `triggeredCandidates`。页面默认只展示主反馈；候选列表用于解释与调试，不是多个同时展示的批评项。

所有默认阈值均来自 `src/config/rules.json`，属于“原型待验证数值”，不代表官方标准、真实平衡结论或经过真实玩家数据验证的判断线。

## 默认候选规则

| 规则 | 维度 | 默认触发条件 |
| --- | --- | --- |
| `FIRST_CHASE_TOO_LONG` | 追击 | `firstChaseDuration > 75000ms`，且首个追逐是非删失完整区间。只计算同一 `chaseId` 的正式追逐起止。 |
| `LATE_FIRST_ELIMINATION` | 减员结果 | `firstEliminationGeneratorsRemaining <= 1`；或者对局正常结束且 `totalEliminations = 0`。不检查挂钩分布，不区分献祭、处决和流血减员的打法优劣。 |
| `ENGAGEMENT_GAP_TOO_LONG` | 接敌 | `averageChaseGap > 30000ms`，且指标达到最低样本要求。开局空窗和后续完整追逐间空窗使用同一平均口径。 |

`LATE_FIRST_ELIMINATION` 使用发电机修理目标而不是绝对时间，让追逐施压和发电机控场两种方向能够在同一结果口径下比较。提示文案不得要求针对某名逃生者，也不得把挂钩集中描述为更优策略。

## 选择唯一主断点

所有触发候选使用以下稳定顺序比较：

1. 证据充分性，由有效样本量相对最低样本要求确定。
2. 相对阈值偏离。
3. 规则配置中的置信度。
4. 配置中的固定规则优先级。
5. 稳定的规则 ID 字典序。

默认固定优先级为：首次追逐、有效接敌空窗、首次减员结果。它只在此前证据完全相同时决定结果。第一版不再提供新手/进阶切换，避免在阈值和分析标准并未经过分层数据校准时制造无意义的选项。

## 证据与输出

每条触发反馈包含：

- `ruleId`、标题、维度和内部严重度；
- 一条限定语气的说明和一个下一局可练习目标；
- 最多三个直接相关指标；
- 每项指标的状态、数值、单位、比较运算与阈值；
- 合并后的 `evidenceEventIds`，供页面定位时间线证据。

内部严重度不得转换为 S/A/B 评级或总分。

## 无明确断点

以下情况返回 `no_clear_breakpoint`：

- 没有规则超过配置阈值；
- 规则所需指标不可用或未达到最低样本；
- 规则配置无效；
- 异常结束日志没有足够的最终减员证据。

无明确断点不是“表现优秀”的评级，只表示当前日志没有提供足够稳定的主要归因。

## 实现边界与尚未覆盖

- 高进度发电机指标保留在详细分析中，但第一版不再单独形成主要候选。
- 规则结果已接入 React 页面和阈值编辑界面；阈值变化会通过统一分析入口重新计算。
- 规则配置目前是项目内 JSON；UI 修改仍需通过同一运行时配置校验。
- 规则阈值尚未通过真实玩家样本回测或校准。
- 当前只使用一套固定分析标准；尚未基于不同水平玩家数据校准分层阈值。
